# 项目路线图

## 1. 路线原则

首版不要同时铺开卡密、直充、代充三条履约链路。优先完成卡密商品的可交易闭环，把订单状态机、支付幂等、退款幂等、库存强一致、后台权限和审计日志打牢。

直充和代充在 MVP 阶段只完成模型预留和后台入口规划，避免早期复杂上游路由拖慢核心交易链路。

## 2. 阶段规划

### Phase 0：工程基础

目标：建立可持续开发的基础结构。

交付项：

- Vue3 + Vite 用户端项目。
- Vue3 + Vite 后台项目。
- Spring Boot 后端项目。
- MySQL 8 初始化脚本与迁移策略。
- Redis 连接与基础缓存封装。
- 登录认证、基础 RBAC、统一错误码、统一响应结构。
- 开发、测试、生产环境配置隔离。

验收：

- 三端可本地启动。
- 后端可连接 MySQL 和 Redis。
- 后台账号可登录并读取菜单权限。

### Phase 1：卡密 MVP 闭环

目标：用户能完成从商品浏览到支付发卡的完整链路。

交付项：

- 五级分类管理。
- 平台标签管理。
- 商品和 SKU 管理。
- 用户组可见性与价格规则。
- 卡密批量导入、去重、库存状态管理。
- 用户端商品列表、平台过滤、商品详情。
- 订单创建、支付单创建、支付回调。
- 支付成功后自动发卡。
- 订单列表、订单详情、卡密展示。
- 后台订单查询和订单导出。
- 审计日志。

关键技术任务：

- 订单状态机落库并限制非法流转。
- 支付回调幂等。
- 卡密库存事务内锁定和发放。
- 支付成功但发卡失败的补偿任务。

验收：

- 重复支付回调不会重复发卡。
- 并发购买不会超卖。
- 后台可追踪每笔订单、支付、发卡记录。
- 订单导出异步执行，不阻塞后台查询。

### Phase 2：支付、退款与运营增强

目标：提升财务处理、风控和运营效率。

交付项：

- 支付主动查询。
- 退款单管理和退款幂等。
- 卡密库存预警。
- 短信通知。
- 更细粒度后台权限。
- 审计日志查询与导出。
- 用户组价格策略增强。
- API 用户基础管理。

验收：

- 回调丢失时可通过支付查询修复状态。
- 同一退款请求重复提交不会重复退款。
- 库存低于阈值可通知后台。
- 敏感操作可审计到操作人和前后值。

### Phase 3：会员 API

目标：支持外部会员系统或代理客户通过 API 接入。

交付项：

- accessKey/secretKey 管理。
- API 签名、nonce、防重放。
- 商品查询 API。
- 创建订单 API。
- 查询订单 API。
- API 限流、IP 白名单、调用日志。
- API 用户账单和导出。

验收：

- API 调用可按用户隔离权限和商品范围。
- 重复请求不会重复创建有效订单。
- 调用失败有明确错误码和日志可查。

### Phase 4：直充履约

目标：接入上游 API 并支持多渠道路由。

交付项：

- 直充任务表。
- 上游渠道配置。
- 上游 API 适配器接口。
- 渠道路由规则。
- 失败重试和人工介入。
- 上游请求日志。
- 上游监控和告警。

路由策略演进：

- V1：固定渠道。
- V2：按商品和平台配置渠道优先级。
- V3：结合成功率、延迟、成本、限流进行动态路由。

验收：

- 支付成功后创建直充任务。
- 任务状态可追踪、可重试、可人工处理。
- 上游异常不会阻塞订单主流程。

### Phase 5：代充和高级运营

目标：支持人工履约和更复杂的运营流程。

交付项：

- 代充工单。
- 人工领取、分配、备注、完成、驳回。
- 处理时效统计。
- 客服协作记录。
- 订单售后处理。
- 多维运营报表。

验收：

- 人工处理链路完整可追踪。
- 每次人工操作都有审计日志。
- 用户能看到明确的处理进度。

## 3. 优先级列表

P0：

- 卡密商品闭环。
- 订单状态机。
- 支付回调幂等。
- 卡密库存强一致。
- 后台 RBAC。
- 审计日志。

P1：

- 退款幂等。
- 订单异步导出。
- 短信通知。
- 库存预警。
- 支付主动查询。

P2：

- 会员 API。
- 直充任务表。
- 上游监控。
- 多渠道路由。
- 代充工单。

## 4. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 卡密超卖 | 资金和履约事故 | MySQL 事务、行级锁、唯一约束、补偿任务 |
| 重复支付回调 | 重复发卡或状态错乱 | 支付单幂等键、状态机限制、发卡前检查 |
| 退款重复提交 | 财务损失 | 退款单唯一业务号、回调幂等、状态限制 |
| 上游不稳定 | 直充失败率高 | 任务表隔离、重试、人工介入、渠道监控 |
| 后台误操作 | 数据损坏 | RBAC、二次确认、审计日志、必要时软删除 |
| 大数据导出阻塞 | 后台不可用 | 异步导出、分页查询、文件过期清理 |

## 5. 暂缓事项

以下能力不进入 MVP：

- 动态多渠道直充路由。
- 复杂会员结算体系。
- 多租户。
- 自动售后仲裁。
- 高级营销活动。
- 全量实时数仓。
